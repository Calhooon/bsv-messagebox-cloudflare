# M9 #52 — TS authsocket wire-compat probe results

## TL;DR

An unmodified `@bsv/authsocket-client` (socket.io v4 under the hood) **cannot connect** to our raw-WS `/ws` endpoint. This holds for every published 1.x and 2.x client version, in both local and deployed-prod environments, with or without forged BRC-31 headers. The failure has TWO layers — auth gate AND protocol gate — both must be defeated for a TS client to work, and even defeating both still leaves the Engine.IO ↔ raw-WS framing mismatch.

## Verdict

TS authsocket clients require **two independent migrations** to talk to `/ws`:

1. **Auth migration**: sign every transport request with BRC-31 headers (socket.io has no native concept of this; would need a custom `Manager` / `Transport` subclass).
2. **Protocol migration**: replace the entire socket.io / Engine.IO stack with a raw-WS client speaking our `{event, data}` JSON envelope.

The cost-benefit favors writing a new client over shimming the old one (see "Shim feasibility" below).

## Probe matrix

The probe runs across two dimensions:

- **Versions tested**: `@bsv/authsocket-client` 1.0.13, 2.0.1, 2.0.2 (the latest of the 1.x line + both 2.x releases). 1.0.0 is omitted: it has a pre-existing import bug — the published package references `dist/cjs/mod.client.js` which doesn't exist in the tarball. Confirmed with `npm view @bsv/authsocket-client@1.0.0 dist.tarball`. Not a wire-compat finding; a packaging bug that predates M9.
- **Auth modes**: `unsigned` (no `extraHeaders`) and `signed` (real BRC-31 headers attached via socket.io's `extraHeaders` option, generated by `sign_for_probe.py` using the same `build_signed_ws_headers` helper that powers `tests/e2e_ws_lifecycle.py`).
- **Targets**: local `wrangler dev` (`http://localhost:8787`) AND deployed prod (`https://rust-message-box.dev-a3e.workers.dev`).

That's 3 versions × 2 modes × 2 targets = **12 probe runs**, all logged as JSONL and checked into `tests/wire_compat_probe/` for replay.

## Findings — uniform across the matrix

| Mode → | unsigned | signed (valid BRC-31) |
|---|---|---|
| **Local 1.0.13** | 401 UNAUTHORIZED | 404 ERR_NOT_FOUND |
| **Local 2.0.1** | 401 UNAUTHORIZED | 404 ERR_NOT_FOUND |
| **Local 2.0.2** | 401 UNAUTHORIZED | 404 ERR_NOT_FOUND |
| **Prod 1.0.13** | 401 UNAUTHORIZED | 404 ERR_NOT_FOUND |
| **Prod 2.0.1** | 401 UNAUTHORIZED | 404 ERR_NOT_FOUND |
| **Prod 2.0.2** | 401 UNAUTHORIZED | 404 ERR_NOT_FOUND |

In every cell, socket.io retries internally for the full 6 s timeout and never reaches `connected = true`. **Local and prod responses are byte-identical** — confirming there's no CDN-layer / rate-limit / origin-header difference between the two.

## What each layer means

- **401 UNAUTHORIZED** (unsigned): our BRC-31 middleware (M9 #40) runs before route dispatch on every HTTP request. The Engine.IO polling GET (`GET /socket.io/?EIO=4&transport=polling&t=...`) carries no `x-bsv-auth-*` headers, so the middleware short-circuits with the parity wire shape `{"status":"error","code":"UNAUTHORIZED","message":"Mutual-authentication failed!"}`. Same shape verified by `tests/e2e_ws_auth.py`. Same shape from commit `41f81b2`. Parity holds across upgrade requests, polling probes, and any other unauthenticated HTTP request.
- **404 ERR_NOT_FOUND** (signed): with valid BRC-31 headers attached via `extraHeaders`, the polling GET passes auth — but our route table has no `/socket.io/` entry, so the request falls through to the catch-all 404. The protocol gate is independent of (and stricter than) the auth gate. Even if a TS client somehow wired BRC-31 into socket.io's transport layer, this would still 404.

The signed probe is the meaningful finding because it isolates the auth gate from the protocol gate. Auth is the OUTER barrier; protocol is the INNER barrier. **Both must be defeated for a connection.**

## What a TS migration would actually need

To talk to `/ws`, a TS client would need:

1. A raw-WS connection (not socket.io / Engine.IO) — drop the polling-then-upgrade dance, drop Engine.IO packet framing.
2. BRC-31 headers on the upgrade GET — generated client-side using `@bsv/sdk`. The Python equivalent is `tests/e2e_ws_lifecycle.py::build_signed_ws_headers`; a TypeScript port would call the same `getOrCreateSession` / `generateNonce` / `createSignature` / `buildAuthHeaders` primitives in TS.
3. The `{event, data}` JSON envelope on every WS frame — same event names as authsocket emits today (`connected`, `joinedRoom`, `sendMessage`, etc.; see `CLAUDE.md` "WebSocket Surface (M9)") but plain JSON over raw frames.

Both changes are small in isolation. Together they amount to "rewrite the transport layer" — at which point the natural home is a new client module on the `@bsv/sdk` side rather than patches to `@bsv/authsocket-client`.

## Shim feasibility

A "drop-in shim Worker" that accepts socket.io connections and forwards them to `/ws` is theoretically possible but **not recommended**. It would need:

- A socket.io / Engine.IO server impl in Rust+WASM on Cloudflare Workers (no off-the-shelf option exists; you'd be writing one).
- Stateful translation between Engine.IO packet types and our `{event, data}` envelope — Engine.IO has 7 packet types (`open`, `close`, `ping`, `pong`, `message`, `upgrade`, `noop`); only `message` maps to our envelope.
- Per-connection BRC-31 session derivation from the client identity (which socket.io has no native concept of).

Estimated effort: several weeks. Not justified unless there's a specific TS-only consumer that won't migrate.

## Reproducing

All artifacts live in `tests/wire_compat_probe/`:

```bash
cd tests/wire_compat_probe
npm install              # installs 4 versions side-by-side via npm aliases
node probe.mjs           # original 5-URL, single-version probe (RESULTS.md v1)
node probe-aggressive.mjs                        # 3 versions × unsigned, local
TARGET_URL=https://rust-message-box.dev-a3e.workers.dev node probe-aggressive.mjs
                                                 # 3 versions × unsigned, prod
HEADERS=$(python3 sign_for_probe.py 'http://localhost:8787/socket.io/?EIO=4&transport=polling&t=probe')
BRC31_HEADERS_JSON="$HEADERS" node probe-aggressive.mjs
                                                 # 3 versions × signed, local
TARGET_URL=https://rust-message-box.dev-a3e.workers.dev \
  HEADERS=$(TARGET_URL=https://rust-message-box.dev-a3e.workers.dev python3 sign_for_probe.py 'https://rust-message-box.dev-a3e.workers.dev/socket.io/?EIO=4&transport=polling&t=probe') \
  BRC31_HEADERS_JSON="$HEADERS" node probe-aggressive.mjs
                                                 # 3 versions × signed, prod
```

Output is JSONL, one event per line. Each `raw-fetch` event captures the literal HTTP request and response so the failure mode is independently auditable. Local and prod runs were captured in `/tmp/probe_*.jsonl` during M9 #52 close-out.

## Coverage

This probe characterizes the gap definitively for the matrix it covers. No outstanding asterisks.
